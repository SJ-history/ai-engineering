# AI Engineering Standard

> 개인용 **AI 엔지니어링 표준**. 어떤 조직·언어·환경(IntelliJ+Claude, VS Code+Copilot,
> Cursor 단독/병행 등)에 가더라도 코딩 에이전트를 **신뢰 가능하고 검증 가능한 방식**으로
> 운용하기 위한 원칙·플레이북·템플릿 모음.

이 저장소의 규칙 체계는 **파일 하나**로 끝난다. 도구별 규칙 파일을 만들지 않는다.
도구는 계속 바뀌지만, **하네스 엔지니어링(Harness)** 과 **루프 엔지니어링(Loop)** 의 원칙은 이식 가능하다.

## 단일 규칙 파일 원칙 (One Rules File)

**규칙의 정본은 `templates/AGENTS.md` 하나다.** 새 프로젝트에 가면 이 파일을 레포 루트에
복사해 채우는 것으로 세팅이 끝난다.

`AGENTS.md`는 특정 회사 제품이 아니라 **업계 공통 표준 파일명**이고, Cursor·GitHub Copilot·
Claude 등 주요 코딩 에이전트가 레포 루트의 `AGENTS.md`를 자동 인식한다.

혹시 쓰는 에이전트가 `AGENTS.md`를 자동으로 읽지 못하면(예: 일부 버전의 Claude는 `CLAUDE.md`만 읽음),
**규칙을 복제하지 말고 한 줄짜리 포인터 파일**만 만든다:

```markdown
<!-- CLAUDE.md — 내용은 이 한 줄이 전부 -->
이 프로젝트의 모든 규칙·명령·검증 신호는 루트 `AGENTS.md`를 따른다.
```

규칙 본문은 언제나 한 곳에만 존재한다. 이러면 어떤 환경에서 clone 받아도 관리 지점이 하나다.

## 왜 필요한가 (Problem Statement)

코딩 에이전트의 실패는 대부분 모델의 지능 부족이 아니라 **운용 환경의 부재**에서 온다.

- 에이전트가 프로젝트 맥락을 몰라 엉뚱한 관습으로 코드를 짠다. → **하네스 결함**
- 에이전트가 "완료했다"고 하지만 실제로는 검증되지 않았다. → **루프 결함**

이 표준의 목적은 **모델·도구가 아무리 바뀌어도 결과 품질의 하한선을 보장하는 운용 체계**를 만드는 것이다.

## 핵심 프레임워크

| 축 | 정의 | 성격 | 핵심 질문 |
|----|------|------|----------|
| **하네스(Harness)** | 에이전트가 작동하는 환경·제약·인터페이스 | 정적(Static) | "에이전트가 올바른 맥락과 도구를 갖고 있는가?" |
| **루프(Loop)** | 계획→실행→관찰→검증→교정의 반복 사이클 | 동적(Dynamic) | "에이전트가 현실 신호로 스스로를 교정하는가?" |

> **하네스는 경기장을 만드는 일, 루프는 경기장 위에서 방향을 계속 교정하는 일.**
> 둘 중 하나라도 부실하면 에이전트는 "그럴듯하지만 검증되지 않은 결과"를 낸다.

## 새 환경 3단계 세팅 (모든 IDE·에이전트 공통)

1. **이 저장소를 clone** 한다.
2. `templates/AGENTS.md`를 새 프로젝트 **레포 루트에 복사**하고 빈칸을 채운다.
   (에이전트가 `AGENTS.md`를 못 읽는 경우에만 위의 한 줄 포인터 파일 추가)
3. 작업할 때 `templates/task-loop-checklist.md`를 곁에 두고 루프를 돈다.

IDE가 IntelliJ든 VS Code든 Cursor든, 에이전트가 Claude든 Copilot이든 내장 에이전트든 **절차는 동일하다.**

## 사용 관점 (Reading Paths)

- **처음 배우는 사람**: `docs/01-framework.md` → `02` → `03` 순으로 이론을 잡고, `templates/`로 실습.
- **새 프로젝트에 방금 투입된 사람**: `docs/04-onboarding-playbook.md`를 그대로 실행.
- **현재 환경을 점검하려는 사람**: `docs/05-maturity-model.md`로 자가진단 후 부족한 기둥을 보강.
- **바로 세팅만 하려는 사람**: 위의 "새 환경 3단계 세팅"이 전부다.

## 설계 원칙 (요약)

1. **단일 규칙 파일(One Rules File)** — 규칙 정본은 `AGENTS.md` 하나. 도구별 규칙 파일을 만들지 않는다.
2. **검증 우선(Verification-first)** — 검증 신호가 없는 작업은 시작하지 않는다.
3. **작은 루프(Small loops)** — 변경 단위를 작게, 피드백 주기를 짧게.
4. **살아있는 문서(Living docs)** — 겪은 실패를 규칙으로 환원해 하네스를 강화한다.
5. **이식성(Portability)** — 조직·IDE·에이전트가 바뀌어도 이 저장소는 그대로 재사용된다.
