# 03. 루프 엔지니어링 (심화)

> 루프 = 에이전트가 도는 동적 피드백 사이클. 루프의 본질은
> **"모델의 추측을 현실 신호로 계속 교정하는 것"** 이다.

## 에이전트 루프의 구조

```
   ┌──────────────────────────────────────────────────┐
   │                                                    ▼
[계획] ──▶ [실행] ──▶ [관찰] ──▶ [검증] ──▶ 통과? ──예──▶ [커밋/다음]
 Plan      Act      Observe    Verify      │
   ▲                                       아니오
   └──────────── [교정/롤백] ◀──────────────┘
                  (N회 실패 시 → 에스컬레이션)
```

각 단계는 다음 5개 요소로 강화된다.

---

## 요소 1: 검증 신호 — 그라운드 트루스 (가장 중요)

에이전트가 "완료"를 선언하는 근거는 **자기 보고가 아니라 기계 판정**이어야 한다.

### 검증 신호의 위계 (신뢰도 순)
| 등급 | 신호 | 신뢰도 | 예 |
|------|------|--------|-----|
| A | 실행 결과 | 최상 | 실제 요청/응답, DB 상태, 스크린샷, E2E |
| B | 테스트 | 높음 | 단위·통합 테스트 통과 |
| C | 타입/컴파일 | 중간 | `tsc`, 빌드 성공 |
| D | 정적 분석 | 보조 | 린트, 포매터 |

> **원칙**: 최소 1개 이상의 신호를 루프에 넣는다. 등급이 높을수록 좋다.
> 신호가 없으면 그 루프는 "모델이 자기 답을 자기가 채점하는" 가짜 루프다.

### 검증 신호 설계는 별도 산출물로
→ `templates/verification-matrix.md`에서 작업 유형별로 어떤 신호를 붙일지 매핑한다.

---

## 요소 2: 루프 주기 최소화 (Short Feedback Loop)

- 큰 작업 한 방 ❌ → 작은 변경 → 즉시 검증 → 다음 ⭕
- 루프 한 바퀴가 짧을수록:
  - 실패 지점이 국소화된다
  - 롤백 비용이 싸다
  - 에이전트의 맥락 이탈이 줄어든다

### 분해 예시
```
❌ "결제 기능 전체 구현"
✅ ① 결제 요청 스키마 + 검증  → 테스트
   ② 결제 게이트웨이 호출부   → 통합 테스트
   ③ 실패/재시도 처리         → 테스트
   ④ UI 연결                  → 실행 확인
```

---

## 요소 3: 종료 조건과 에스컬레이션 (Termination)

자율 루프에는 반드시 **멈추는 규칙**이 있어야 한다.

- **재시도 상한**: "같은 문제 N회(권장 3회) 실패 시 중단"
- **에스컬레이션**: 중단 시 "무엇을 시도했고 각각 왜 실패했는지"를 정리해 사람에게 보고
- **예산 한계**: 시간·토큰·비용 상한 → 무한 루프·자원 낭비 방지
- **불확실성 게이트**: 확신이 없으면 실행 전에 질문 (파괴적 작업일수록 강하게)

---

## 요소 4: 자기 교정 (Self-Correction)

실패했을 때 되돌아가 다른 경로를 시도하는 능력.

- **롤백**: git으로 마지막 정상 지점 복귀 (가드레일과 연동)
- **재계획**: 같은 방법 반복 ❌ → 실패 원인 분석 후 대안 수립
- **가설 검증**: "왜 실패했는가"에 대한 가설을 세우고 최소 변경으로 검증

> 나쁜 루프: 같은 수정을 계속 반복. 좋은 루프: 실패에서 정보를 얻어 경로를 바꾼다.

---

## 요소 5: 평가 (Evaluation)

"이 루프가 실제로 좋은 결과를 내는가"를 반복 측정한다.

- **골든 케이스**: 대표 작업의 기대 결과를 고정해두고 회귀 확인
- **루프 지표**: 검증 통과율, 평균 시도 횟수, 롤백 빈도, 에스컬레이션 비율
- **개선 환류**: 반복 실패 패턴 → 하네스의 규칙(금지사항)으로 환원 (`02` 기둥1·P5)

평가가 없으면 루프는 "돌아가는 것 같지만 품질을 모르는" 블랙박스가 된다.

---

## 루프 운영 체크리스트 (요약)

**준비**
- [ ] 완료 기준을 한 문장으로 정의했는가
- [ ] 검증 신호(A~D 중 최소 1개)를 확보했는가
- [ ] 브랜치를 분기했는가 (복구 가능성)

**반복**
- [ ] 작업을 작은 단위로 쪼갰는가
- [ ] 한 단위만 실행 → 검증했는가
- [ ] 통과 → 커밋 / 실패 → 원인 분석 후 재시도
- [ ] N회 실패 → 중단하고 보고

**마무리**
- [ ] 전체 검증 통과, 디버그 잔재 제거
- [ ] 반복된 실패를 규칙으로 환원했는가

→ 실전 체크리스트: `templates/task-loop-checklist.md`
