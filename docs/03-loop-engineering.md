# 03. 루프 엔지니어링 (심화)

> 루프 = 에이전트가 도는 동적 피드백 사이클. 루프의 본질은
> **"모델의 추측을 현실 신호로 계속 교정하는 것"** 이다.

## 에이전트 루프의 구조

```
   ┌──────────────────────────────────────────────────┐
   │                                                    ▼
[계획] ──▶ [실행] ──▶ [관찰] ──▶ [검증] ──▶ 통과? ──예──▶ [커밋/다음]
 Plan      Act      Observe    Verify      │
   ▲                                       아니오
   └──────────── [교정/롤백] ◀──────────────┘
                  (N회 실패 시 → 에스컬레이션)
```

각 단계는 다음 5개 요소로 강화된다.

---

## 요소 1: 검증 신호 — 그라운드 트루스 (가장 중요)

에이전트가 "완료"를 선언하는 근거는 **자기 보고가 아니라 기계 판정**이어야 한다.

### 검증 신호의 위계 (신뢰도 순)
| 등급 | 신호 | 신뢰도 | 예 |
|------|------|--------|-----|
| A | 실행 결과 | 최상 | 실제 요청/응답, DB 상태, 스크린샷, E2E |
| B | 테스트 | 높음 | 단위·통합 테스트 통과 |
| C | 타입/컴파일 | 중간 | `tsc`, 빌드 성공 |
| D | 정적 분석 | 보조 | 린트, 포매터 |

> **원칙**: 최소 1개 이상의 신호를 루프에 넣는다. 등급이 높을수록 좋다.
> 신호가 없으면 그 루프는 "모델이 자기 답을 자기가 채점하는" 가짜 루프다.

### 검증 신호 설계는 별도 산출물로
→ `templates/verification-matrix.md`에서 작업 유형별로 어떤 신호를 붙일지 매핑한다.

### 검증자 분리 (Maker ≠ Checker) — 자기 채점 편향 방지
검증 신호가 있어도, **그 신호를 판정하는 주체가 만든 주체와 같으면 자기 편향이 낀다.**
에이전트는 자기 결과를 스스로 채점할 때 후하게 평가하는 경향이 있다.

- **실측 사례**: 같은 산출물을 자기 채점 9.04점, 독립된 다른 에이전트가 채점 7.43점 —
  1.6점 괴리. (개인 런 이력이라 벤치마크로 일반화할 수는 없지만, 경향 자체는 재현되는 패턴이다.)
- **원칙**: 만드는 에이전트(Maker)와 검사하는 에이전트(Checker)를 분리한다.
  가능하면 모델·프롬프트·컨텍스트를 다르게 해서 같은 편향을 공유하지 않게 한다.
- 사람이 최종 리뷰어라도, 이 1차 교차검증이 있으면 사람이 매번 처음부터 검토할 부담이 줄어든다.
- "검증 신호가 있다"와 "그 신호를 신뢰할 수 있다"는 다른 문제다. 후자를 위해 채점자를
  제작자로부터 분리하는 것이 이 요소의 핵심이다.

## 요소 2: 루프 주기 최소화 (Short Feedback Loop)

- 큰 작업 한 방 ❌ → 작은 변경 → 즉시 검증 → 다음 ⭕
- 루프 한 바퀴가 짧을수록:
  - 실패 지점이 국소화된다
  - 롤백 비용이 싸다
  - 에이전트의 맥락 이탈이 줄어든다

### 분해 예시
```
❌ "결제 기능 전체 구현"
✅ ① 결제 요청 스키마 + 검증  → 테스트
   ② 결제 게이트웨이 호출부   → 통합 테스트
   ③ 실패/재시도 처리         → 테스트
   ④ UI 연결                  → 실행 확인
```

---

## 요소 3: 종료 조건과 에스컬레이션 (Termination)

자율 루프에는 반드시 **멈추는 규칙**이 있어야 한다.

- **재시도 상한**: "같은 문제 N회(권장 3회) 실패 시 중단"
- **에스컬레이션**: 중단 시 "무엇을 시도했고 각각 왜 실패했는지"를 정리해 사람에게 보고
- **예산 한계**: 시간·토큰·비용 상한 → 무한 루프·자원 낭비 방지
- **불확실성 게이트**: 확신이 없으면 실행 전에 질문 (파괴적 작업일수록 강하게)

---

## 요소 4: 자기 교정 (Self-Correction)

실패했을 때 되돌아가 다른 경로를 시도하는 능력.

- **롤백**: git으로 마지막 정상 지점 복귀 (가드레일과 연동)
- **재계획**: 같은 방법 반복 ❌ → 실패 원인 분석 후 대안 수립
- **가설 검증**: "왜 실패했는가"에 대한 가설을 세우고 최소 변경으로 검증

> 나쁜 루프: 같은 수정을 계속 반복. 좋은 루프: 실패에서 정보를 얻어 경로를 바꾼다.

---

## 요소 5: 평가 (Evaluation)

"이 루프가 실제로 좋은 결과를 내는가"를 반복 측정한다.

- **골든 케이스**: 대표 작업의 기대 결과를 고정해두고 회귀 확인
- **루프 지표**: 검증 통과율, 평균 시도 횟수, 롤백 빈도, 에스컬레이션 비율
- **개선 환류**: 반복 실패 패턴 → 하네스의 규칙(금지사항)으로 환원 (`02` 기둥1·P5)

평가가 없으면 루프는 "돌아가는 것 같지만 품질을 모르는" 블랙박스가 된다.

---

## 요소 6: 루프의 구현 부품 (5+1 Components)

> 위 다섯 요소가 "루프가 갖춰야 할 성질"이라면, 여기부터는 그 성질을 **실제로 무엇으로
> 구현하는가**다. 도구(Claude Code, Codex 등)가 바뀌어도 부품의 역할은 이식된다.

| 부품 | 역할 | 없으면 생기는 문제 |
|------|------|--------------------|
| ① 오토메이션 (Automation) | 정해진 주기·조건에 사람 개입 없이 루프를 시작 (cron, hooks, `/loop`류 명령) | 매번 사람이 직접 트리거해야 함 → 루프가 아니라 대화 |
| ② 워크트리 (Worktree) | 작업마다 격리된 작업 공간을 줘서 동시 작업 충돌 방지 | 여러 작업이 같은 브랜치에서 서로의 변경을 덮어씀 |
| ③ 스킬 (Skill) | 반복 설명이 필요 없는 프로젝트 지식을 파일로 코드화 (`AGENTS.md`의 하위 계층) | 매 세션 같은 맥락을 처음부터 재설명해야 함 |
| ④ 플러그인·커넥터 (Connector) | 외부 도구(이슈 트래커·DB·알림)와 연결 (MCP 등 표준 프로토콜) | 결과를 사람이 수동으로 옮겨 적어야 함 |
| ⑤ 서브에이전트 (Sub-agent) | 만드는 에이전트와 검사하는 에이전트 분리 (→ 요소1 "검증자 분리") | 자기 채점 편향이 검증 신호를 오염시킴 |
| ⑥ 상태 파일 (State/Memory) | 세션이 끝나도 남는 진행 기록. "에이전트는 잊어도 저장소는 잊지 않는다" | 다음 세션에서 어디까지 했는지 다시 추론해야 함 |

### 산출물
- `templates/progress-template.md` — 상태 파일(⑥)의 최소 형태. 시도한 것 · 통과한 것 ·
  남은 것을 세션 간 이어지는 마크다운 한 장으로 기록한다.

> 초심자는 6개를 한 번에 갖추려 하지 말 것. `templates/AGENTS.md`(③의 최소형)와
> `templates/progress-template.md`(⑥) 두 개부터 시작해도 다음 세션 설명 비용이 눈에 띄게 준다.

## 위험 신호: 인지적 항복 (Cognitive Surrender)

> 루프가 매끄러워질수록 사람이 "이해하기"를 포기하고 싶은 유혹이 커진다.
> 이것은 개별 요소의 결함이 아니라 **하네스·루프 전체를 무력화하는 가장 큰 리스크**다.

- **착각**: 루프는 사람을 일에서 빼준다. → 아니다. 루프는 반복 작업만 빼줄 뿐, 검증·이해의
  책임은 그대로 사람 몫이다(요소1, P2).
- **경고 신호**: 산출물이 빨리 나올수록, 읽지 않고 승인하는 습관이 생긴다. "생각하기 자체를
  기계에 넘기는 순간"이 루프가 숙련의 도구가 아니라 **이해를 피하는 핑계**가 되는 지점이다.
- **대응**: 자율성(하네스 P4 경계, 이 문서 요소3 종료조건)을 높인 만큼, 사람의 리뷰·이해
  노력도 비례해서 유지해야 한다. 루프는 그 차이를 모르지만, 사람은 안다.

> 한 문장: **"루프를 만들되, '시작' 버튼만 누르는 사람이 아니라 끝까지 엔지니어로 남을
> 작정으로 만든다."**

## 루프 운영 체크리스트 (요약)

**준비**
- [ ] 완료 기준을 한 문장으로 정의했는가
- [ ] 검증 신호(A~D 중 최소 1개)를 확보했는가
- [ ] 브랜치를 분기했는가 (복구 가능성)

**반복**
- [ ] 작업을 작은 단위로 쪼갰는가
- [ ] 한 단위만 실행 → 검증했는가
- [ ] 위험도가 높은 단위는 만든 에이전트와 다른 에이전트/사람이 교차 검증했는가 (검증자 분리)
- [ ] 통과 → 커밋 / 실패 → 원인 분석 후 재시도
- [ ] N회 실패 → 중단하고 보고

**마무리**
- [ ] 전체 검증 통과, 디버그 잔재 제거
- [ ] 결과물을 읽지 않고 승인하지 않았는가 (인지적 항복 점검)
- [ ] 반복된 실패를 규칙으로 환원했는가
- [ ] 상태 파일(`progress-template.md`)에 이번 세션의 진행 상황을 남겼는가

→ 실전 체크리스트: `templates/task-loop-checklist.md`
