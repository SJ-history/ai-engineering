# 05. 성숙도 모델 (Maturity Model) 과 자가진단

> 현재 환경의 하네스·루프 수준을 객관적으로 측정하고, 다음 목표를 설정하기 위한 척도.
> "우리 팀/이 프로젝트는 지금 어느 레벨인가?"에 답하기 위한 도구.

## 성숙도 레벨 (L0 → L4)

| 레벨 | 이름 | 특징 | 리스크 |
|------|------|------|--------|
| **L0** | Ad-hoc (임기응변) | 컨텍스트·검증 없이 그때그때 프롬프트로 코딩 | 결과 편차 극심, 가짜 완료 빈발 |
| **L1** | Repeatable (반복 가능) | `AGENTS.md` 등 기본 컨텍스트 존재 | 검증은 여전히 수동/누락 |
| **L2** | Verified (검증 도입) | 루프에 검증 신호(테스트/빌드) 연결 | 신호 커버리지 불균일 |
| **L3** | Guarded (통제 확립) | 가드레일·종료조건·권한 경계 확립 | 개선 환류가 비체계적 |
| **L4** | Optimizing (최적화) | 평가 지표로 루프를 측정·개선, 규칙 자동 환류 | (지속적 유지가 과제) |

각 레벨은 아래 레벨을 포함한다(누적적).

---

## 두 축 × 5기둥 진단 매트릭스

각 항목을 0(없음)/1(부분)/2(확립)로 채점한다.

### 하네스 축
| 기둥 | 진단 질문 | 점수(0–2) |
|------|-----------|:---:|
| 컨텍스트 | 프로젝트 관습·금지사항이 문서화되고 에이전트가 참조하는가 | |
| 도구 | 핵심 명령이 동작하고 에러가 행동 가능한가 | |
| 가드레일 | 브랜치·권한 경계·격리로 안전하게 되돌릴 수 있는가 | |
| 관찰가능성 | 에이전트 행동·근거를 추적할 수 있는가 | |
| 인터페이스 | 지시·보고 형식이 표준화됐는가 | |

### 루프 축
| 요소 | 진단 질문 | 점수(0–2) |
|------|-----------|:---:|
| 검증 신호 | 작업에 그라운드 트루스 신호가 붙어 있는가 | |
| 루프 주기 | 작업을 작게 쪼개 짧은 주기로 도는가 | |
| 종료조건 | 재시도 상한·에스컬레이션 규칙이 있는가 | |
| 자기교정 | 실패 시 롤백·재계획 경로가 있는가 | |
| 평가 | 루프 품질을 지표로 측정·개선하는가 | |

### 레벨 판정 가이드 (총 20점)
- **0–5점**: L0 — 기둥부터 세워라. `04-onboarding-playbook.md` Phase 2 착수.
- **6–10점**: L1 — 검증 신호를 붙여라. `03` 요소1, `verification-matrix.md`.
- **11–14점**: L2 — 가드레일·종료조건을 확립하라. `02` 기둥3, `03` 요소3.
- **15–18점**: L3 — 평가 지표를 도입하고 규칙 환류를 자동화하라. `03` 요소5.
- **19–20점**: L4 — 유지·전파. 다른 프로젝트에 표준을 확산하라.

---

## 레벨업 로드맵

```
L0 → L1 : AGENTS.md 작성 (컨텍스트 확보)
L1 → L2 : 검증 신호 연결 (테스트/빌드/실행을 루프에)
L2 → L3 : 가드레일 + 종료조건 (안전한 자율성)
L3 → L4 : 평가 지표 + 규칙 자동 환류 (측정 기반 개선)
```

> 실무 지침: **한 번에 한 레벨씩**. 검증(L2) 없이 자율성(L3)을 높이면
> "빠르게 잘못된 방향으로 가는" 최악의 조합이 된다. 검증 → 통제 → 최적화 순서를 지켜라.
