# 01. 프레임워크와 설계 원칙

> 이 문서는 AI 엔지니어링 표준의 이론적 토대다. 나머지 모든 문서와 템플릿은 여기서 파생된다.

## 1. 문제 정의: 에이전트는 왜 길을 잃는가

코딩 에이전트(LLM 기반)의 실패는 세 가지 층위에서 발생한다.

| 층위 | 증상 | 실제 원인 | 대응 축 |
|------|------|-----------|---------|
| 맥락 부족 | 프로젝트 관습을 무시, 엉뚱한 위치에 코드 생성 | 환경이 맥락을 제공하지 않음 | 하네스 |
| 검증 부재 | "완료"라고 하지만 실제로는 깨져 있음 | 현실 신호가 루프에 없음 | 루프 |
| 통제 상실 | 무한 반복, 범위 이탈, 되돌릴 수 없는 손상 | 경계·종료조건 부재 | 하네스+루프 |

**핵심 통찰**: 모델 성능을 높이는 것보다, **운용 체계를 갖추는 것이 결과 품질의 분산을 줄인다.**
좋은 하네스와 루프는 약한 모델의 결과를 끌어올리고, 강한 모델의 결과를 신뢰 가능하게 만든다.

## 2. 두 축의 정의

### 하네스 엔지니어링 (Harness Engineering) — 정적 환경
에이전트가 작동하는 **판(scaffolding)** 을 설계하는 일. 마구(harness)가 말의 힘을 원하는 방향으로만
흐르게 하듯, 에이전트의 능력을 목표에 정렬시킨다. → 상세: `02-harness-engineering.md`

### 루프 엔지니어링 (Loop Engineering) — 동적 사이클
에이전트가 도는 **피드백 사이클**을 설계하는 일. 계획→실행→관찰→검증→교정이 스스로 굴러가고,
틀리면 되돌아오게 만든다. → 상세: `03-loop-engineering.md`

```
        ┌───────────────── 하네스 (정적 환경) ─────────────────┐
        │  컨텍스트 · 도구 · 가드레일 · 관찰가능성 · 인터페이스   │
        │                                                      │
        │      ┌──────────── 루프 (동적 사이클) ──────────┐     │
        │      │  계획 → 실행 → 관찰 → 검증 → 교정  ↺      │     │
        │      └───────────────────────────────────────────┘     │
        └────────────────────────────────────────────────────────┘
                 하네스가 판을 깔고, 루프가 그 위에서 돈다
```

## 3. 설계 원칙 (Design Principles)

### P1. 단일 규칙 파일 (One Rules File)
표준은 도구 위에 존재한다. IDE(IntelliJ, VS Code, Cursor)와 에이전트(Claude, Copilot, 내장 에이전트)는
구현체일 뿐이다. 규칙의 정본은 레포 루트의 `AGENTS.md` **하나**로 유지하고, 도구별 규칙 파일을
만들지 않는다. 에이전트가 `AGENTS.md`를 직접 읽지 못하는 경우에만 "정본은 AGENTS.md를 따른다"는
한 줄짜리 포인터 파일을 둔다. → 이직·도구 교체 시 재작성 비용이 0에 수렴한다.

### P2. 검증 우선 (Verification-first)
"자동으로 참/거짓이 판별되는 신호"가 없는 작업은 시작하지 않는다.
모델의 자기 보고는 증거가 아니다. 컴파일·타입·테스트·실행 결과만이 증거다.

### P3. 작은 루프 (Small Loops)
변경 단위를 작게 쪼갤수록 실패 지점이 국소화되고 롤백이 싸진다.
30분짜리 작업을 6개의 5분 루프로 분해하는 것이 기본기다.

### P4. 명시적 경계 (Explicit Boundaries)
에이전트가 무엇을 해도 되고 안 되는지, 언제 멈추고 사람을 불러야 하는지를 사전에 규정한다.
경계가 없으면 자율성은 곧 리스크가 된다.

### P5. 살아있는 문서 (Living Documentation)
겪은 실패는 일회성 수정으로 끝내지 않고 **규칙으로 환원**한다.
"금지사항"과 "체크리스트"가 축적될수록 하네스는 조직 지능이 된다.

### P6. 관찰 가능성 (Observability)
에이전트가 무엇을 왜 했는지 추적 불가능하면 개선도 불가능하다.
행동 로그·근거·도구 호출 기록을 남긴다.

### P7. 협업 경계 (Collaboration Boundary)
이 표준은 **개인용**이지만 실제 작업은 항상 협업 위에서 일어난다. 새 팀에 합류했을 때
이 표준으로 팀의 기존 규칙을 대체하려 하지 않는다.
- 팀에 이미 규칙 파일이 있으면 그것이 정본이고, 이 표준은 그 위의 개인 보충 레이어다.
- "혼자 결정 가능한 것"(작업 방식, 로컬 루프)과 "팀 합의가 필요한 것"(CI, 배포, 리뷰 규칙)을
  항상 구분한다.
- 개인 루프의 검증 결과는 나만 확인하고 끝내지 않고, PR·커밋처럼 팀원도 보는 곳에 근거로
  남긴다. 협업에서 신뢰는 "혼자 잘하는 것"이 아니라 "잘하고 있다는 근거가 보이는 것"에서 온다.

## 4. 컨텍스트 엔지니어링과의 관계

"프롬프트 엔지니어링"은 단발 요청 최적화다. 이 표준이 다루는 것은 그 상위 개념인
**컨텍스트 엔지니어링 + 시스템 엔지니어링**이다:

- 무엇을(맥락) 에이전트에게 **언제** 제공할지 설계 (하네스)
- 에이전트의 출력을 **어떻게 현실로 검증**하고 교정할지 설계 (루프)

프롬프트는 루프의 한 입력일 뿐, 시스템의 전부가 아니다.

## 5. 안티패턴 (반드시 피할 것)

| 안티패턴 | 왜 위험한가 | 올바른 방향 |
|----------|-------------|-------------|
| 모든 맥락을 한 프롬프트에 몰아넣기 | 신호 대비 잡음↑, 길 잃음 | 필요 시 검색·참조하게 (P1, 하네스) |
| 검증 없이 대량 변경 | "가짜 완료"에 속음 | 검증 신호 먼저 확보 (P2) |
| 큰 작업 한 방에 지시 | 실패 지점 불명, 롤백 비쌈 | 작게 쪼개기 (P3) |
| 무제한 자율 실행 | 무한 루프·범위 이탈 | 종료조건·경계 (P4) |
| 실패를 그때만 고치고 잊음 | 같은 실수 반복 | 규칙으로 환원 (P5) |
| 개인 표준을 팀 표준인 것처럼 강요 | 팀 관습과 충돌, 협업 마찰 | 팀 규칙 우선, 개인 표준은 보충 레이어 (P7) |
| 검증 통과만 보고 결과물을 안 읽음 | 인지적 항복, 실력 저하 | 자율성 높인 만큼 리뷰 노력 유지 (`03` 위험 신호) |
