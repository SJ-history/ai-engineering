# 02. 하네스 엔지니어링 (심화)

> 하네스 = 에이전트가 작동하는 정적 환경. 여기서는 하네스를 **5대 기둥**으로 분해하고,
> 각 기둥을 어떤 산출물로 구현하는지 정의한다.

## 하네스의 5대 기둥

```
① 컨텍스트   ② 도구      ③ 가드레일    ④ 관찰가능성   ⑤ 인터페이스
 (Context)   (Tools)    (Guardrails) (Observability) (Interface)
```

에이전트 실패를 진단할 때는 항상 "5개 중 어느 기둥이 무너졌는가"로 환원한다.

---

## 기둥 1: 컨텍스트 (Context)

에이전트가 올바른 판단을 하려면 **프로젝트의 암묵지**를 명시지로 제공해야 한다.

### 무엇을 제공하는가
- 프로젝트의 목적·도메인·주요 사용자
- 아키텍처 개요와 폴더별 책임
- 코딩 관습(네이밍, 계층 구조, 에러 처리 패턴)
- 절대 금지사항(도메인 특유의 지뢰)

### 어떻게 제공하는가 — 계층적 컨텍스트 원칙
> **한 번에 다 주지 말고, 필요할 때 찾아 쓰게 한다.**

전부 프롬프트에 넣으면 잡음이 늘어 오히려 길을 잃는다. 대신:
- **항상 로드**: 짧고 핵심적인 규칙만 (`AGENTS.md`)
- **조건부 로드**: 특정 디렉터리·작업에서만 필요한 세부 규칙
- **검색 기반**: 나머지는 코드베이스에서 직접 찾게

### 기존 팀 규칙과의 병행 (개인 표준 도입 시)
이 컨텍스트는 개인이 들고 다니는 표준이지만, 새 팀에는 이미 `CONTRIBUTING.md`·`CODEOWNERS`
같은 자체 규칙이 있을 수 있다. 이때 원칙은 **대체가 아니라 병행**이다.
- 팀 규칙이 있으면 그것이 정본. `AGENTS.md`는 팀 문서가 다루지 않는 실무 디테일
  (검증 명령, 작업 루프 습관)만 채우는 보충 레이어로 둔다.
- 팀 규칙과 충돌하면 팀 규칙을 따르고 충돌만 기록한다. 조용히 무시하거나 덮어쓰지 않는다.
- 팀에 규칙이 전혀 없으면 "표준"이 아니라 "시험적 제안"으로 공유해 합의를 얻는다.

### 산출물
- `templates/AGENTS.md` — 도구 비종속 프로젝트 컨텍스트 표준

---

## 기둥 2: 도구 (Tools)

에이전트의 손발이다. 도구의 **품질이 자율성의 상한**을 결정한다.

### 좋은 도구의 조건
1. **명확한 계약**: 입력/출력 형식이 예측 가능하다.
2. **행동 가능한 에러**: 실패 시 "무엇이·어디서·어떻게 고칠지"를 담은 메시지를 준다.
3. **결정성(Determinism)**: 같은 입력에 같은 출력. 비결정적 도구는 루프를 오염시킨다.
4. **빠른 피드백**: 느린 도구는 루프 주기를 늘려 에이전트를 헤매게 한다.

### 나쁜 vs 좋은 에러 (핵심)
```
❌ Error: build failed
✅ Error: src/api.ts:42 — Type 'string' is not assignable to 'number'. Did you mean parseInt()?
```
후자는 에이전트가 **사람 개입 없이 스스로 교정**할 수 있다.
→ 이식 가능한 스킬: "사람이 아니라 **에이전트가 읽을** 로그·에러를 설계한다"는 관점.

### 대표 도구 범주
빌드 · 테스트 러너 · 타입체커/린터 · 코드 검색 · 실행/기동 · 배포 · 마이그레이션

---

## 기둥 3: 가드레일 (Guardrails)

자율성을 안전하게 만드는 경계. 가드레일이 있어야 **과감하게** 위임할 수 있다.

### 3-1. 복구 가능성 (Reversibility)
- 작업 전 항상 브랜치 분기 → 손상 시 즉시 원복
- 작은 단위 커밋 → 롤백 지점 촘촘히 확보
- 되돌릴 수 있는 상태는 자율성의 전제 조건이다.

### 3-2. 권한 경계 (Permission Boundaries)
- 수정 가능/불가 영역 명시 (예: 생성 코드는 건드리되 마이그레이션은 확인받기)
- 위험 작업(삭제, force push, 프로덕션 접근, 시크릿)은 명시적 승인 게이트
- 초심자일수록 자동 승인은 끄고 시작

### 3-3. 격리 (Isolation)
- 샌드박스/컨테이너에서 실행해 블라스트 반경 제한
- 시크릿·자격증명은 컨텍스트에 노출하지 않음

---

## 기둥 4: 관찰 가능성 (Observability)

측정할 수 없으면 개선할 수 없다.

- **행동 추적**: 어떤 파일을 왜 수정했고, 어떤 도구를 호출했는가
- **의사결정 근거**: 왜 그 방향을 택했는가 (필요 시 에이전트에게 되물어 추적)
- **루프 지표**: 시도 횟수, 검증 통과율, 롤백 빈도 → 성숙도 측정의 원천

---

## 기둥 5: 인터페이스 (Interface)

사람↔에이전트, 에이전트↔시스템 간 소통 규약.

- **작업 지시 형식**: 목표·완료기준·제약을 구조적으로 전달
- **출력 규약**: 변경 요약·근거·검증 결과를 정해진 형태로 보고하게
- **표준 프로토콜 활용**: `AGENTS.md`, MCP(도구 연결) 등 도구 간 공통 규약 우선 채택
  → 특정 벤더 락인을 피하고 이식성 확보
- **팀에게 보이는 인터페이스**: 검증 결과를 나만 확인하고 끝내지 않는다. PR 설명·커밋
  메시지에 검증 신호의 실행 결과를 근거로 남겨, 팀원이 내 루프 실천을 신뢰할 수 있게 한다.
  개인 루프 습관을 팀 협업의 신뢰 자산으로 바꾸는 지점이 여기다.

---

## 하네스 구축 체크리스트 (요약)

- [ ] **컨텍스트**: `AGENTS.md` 작성, 계층적으로 구성했는가
- [ ] **도구**: 핵심 명령이 실제 동작하고, 에러가 행동 가능한가
- [ ] **가드레일**: 브랜치 전략·권한 경계·격리가 있는가
- [ ] **관찰가능성**: 에이전트 행동·근거를 추적할 수 있는가
- [ ] **인터페이스**: 지시·보고 형식이 표준화됐는가

> 이 5개는 언어·회사·프레임워크와 무관하다. 새 환경에서 이 5개를 세팅하는 루틴만
> 몸에 익히면 어디서든 통한다.
