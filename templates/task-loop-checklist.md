# 작업 루프 체크리스트 (Task Loop Checklist)

> 작업 단위마다 이 절차를 따른다. 이론적 근거: `docs/03-loop-engineering.md`.
> 곁에 열어두고 각 단계를 실제로 확인할 것.

## STAGE A — 준비 (작업 시작 시 1회)
- [ ] **완료 기준**을 한 문장으로 적었는가? → `___________________________`
- [ ] **검증 신호**를 확보했는가? (아래 중 최소 1개, 높을수록 좋음)
- [ ] A. 실행 결과로 확인 가능 (요청/응답·화면·DB·E2E)
- [ ] B. 테스트로 확인 가능 (`___________`)
- [ ] C. 타입/빌드로 확인 가능 (`___________`)
- [ ] D. 린트/정적분석 (보조)
- [ ] **작업 분해**: 큰 목표를 작은 단위로 쪼갰는가?
- [ ] **복구 가능성**: 브랜치를 분기했는가? `git checkout -b feature/<작업>`

## STAGE B — 반복 루프 (단위마다)
```
계획 → 실행(한 단위만) → 관찰 → 검증 → 통과? ── 예 → 커밋 → 다음 단위
                                        └─ 아니오 → 원인 분석 → 교정/롤백 ↺
```
- [ ] 1. 에이전트에게 **한 단위만** 지시했는가?
- [ ] 2. 변경 후 검증 신호로 확인했는가?
- [ ] 3-A. 통과 → 커밋. `git commit -m "<의미 있는 설명>"`
- [ ] 3-B. 실패 → **같은 수정 반복 금지.** 원인 가설 → 최소 변경으로 검증
- [ ] 4. **N회(권장 3회) 실패 → 중단.** 시도 내역·실패 원인 정리 후 판단/질문

## STAGE C — 마무리 (작업 종료 시)
- [ ] 전체 검증(테스트/빌드/실행)이 통과하는가?
- [ ] 디버그 코드·임시 로그·주석을 정리했는가?
- [ ] 시크릿이 커밋에 포함되지 않았는가?
- [ ] **규칙 환류**: 반복된 실수가 있었다면 `AGENTS.md` 금지사항에 추가했는가?
- [ ] 재사용할 만한 개선이면 개인 표준 저장소에 역반영했는가?

## 자주 쓰는 프롬프트 스니펫
- **정찰**: "이 레포를 인수인계받는 시니어 입장에서 구조·명령·진입점·데이터 흐름을 근거와 함께 정리해줘."
- **작게 지시**: "지금은 <이 한 단위>만 구현하고, 끝나면 <검증방법>으로 검증한 결과를 보여줘."
- **막혔을 때**: "지금까지 시도한 것과 각각 실패 원인을 정리하고, 서로 다른 대안 2가지를 제안해줘."
- **검증 강제**: "변경 후 <검증명령>을 실행하고, 통과했다는 증거(출력)를 보여준 다음에 완료라고 말해줘."
