# AGENTS.md — 프로젝트 컨텍스트 표준

> **규칙의 정본(단 하나의 규칙 파일).** 새 레포의 **루트**에 복사하고 `< >`를 채운다.
> `AGENTS.md`는 업계 공통 표준 파일명으로, 주요 코딩 에이전트(Cursor·Copilot·Claude 등)가 자동 인식한다.
> 에이전트가 이 파일을 직접 읽지 못하는 경우에만, "정본은 루트 AGENTS.md를 따른다"는
> 한 줄짜리 포인터 파일(예: CLAUDE.md)을 추가한다. 규칙 본문은 절대 복제하지 않는다.
>
> 원칙: **짧고 핵심적인 규칙만 여기에.** 세부 규칙은 하위 디렉터리의 별도 파일로 분리한다(계층적 컨텍스트).

---

## 0. 팀 표준과의 관계 (정본 우선순위)
> 이 파일은 **개인 표준**이다. 새 팀에 합류했을 때 팀이 이미 가진 규칙을 대체하지 않는다.

- 레포에 이미 `CONTRIBUTING.md`/`CODEOWNERS`/자체 규칙 파일이 있다면 **그것이 정본이고 우선한다.**
  이 파일은 그 위에 얹는 **개인 보충 레이어**로만 채운다(빌드 명령·검증 신호처럼 팀 문서에 없는
  실무 디테일 위주).
- 팀 규칙과 이 파일의 내용이 충돌하면 **팀 규칙을 따르고**, 충돌 사실만 8절에 메모한다
  (팀 규칙을 조용히 무시하거나 이 파일로 덮어쓰지 않는다).
- 혼자 결정해도 되는 것과 팀 합의가 필요한 것을 구분한다:
  - **혼자 결정**: 내 작업 방식(작게 쪼개기, 로컬 검증), 내 브랜치에서의 커밋 단위
  - **팀 합의 필요**: 공유 CI/CD 변경, 브랜치 전략 변경, 배포 게이트, 코드 리뷰 규칙
- 팀에 아무 규칙 파일이 없다면 이 파일을 그대로 도입 제안하되, "내가 정한 표준"이 아니라
  "이런 걸 시험적으로 써보자"는 톤으로 공유한다.

## 7. 절대 금지사항 (Guardrails)
- 확신 없이 대량 파일을 한꺼번에 수정하지 않는다.
- 검증(테스트/빌드)이 깨진 상태로 작업을 종료하지 않는다.
- 시크릿(.env, 키파일)을 커밋하거나 컨텍스트에 노출하지 않는다.
- 다음은 사전 승인 없이 수행 금지: <삭제 / force push / 프로덕션 접근 / 마이그레이션 등>

## 8. 에이전트 작업 규약 (Loop)
1. 코드를 고치기 전에 관련 파일을 읽고 맥락을 파악한다.
2. 큰 작업은 작은 검증 가능한 단위로 쪼갠다.
3. 변경 후 반드시 5절의 검증 신호로 확인하고 결과를 보고한다.
4. 확신이 없으면 고치기 전에 질문한다. 파괴적 작업은 반드시 승인받는다.
5. 같은 문제를 3회 시도해도 실패하면 중단하고, 시도 내역과 실패 원인을 정리해 보고한다.
6. 무엇을·왜 바꿨는지 변경 요약을 남긴다.
7. **검증 결과는 팀원도 보는 곳에 남긴다** — PR 설명·커밋 메시지에 검증 신호(5절)의 실행
   결과를 근거로 첨부한다. 나만 아는 체크리스트로 끝내지 않는다(팀 협업 시 신뢰의 원천).
8. 0절의 팀 규칙과 충돌이 생기면 이 파일이 아니라 팀 규칙을 따르고, 충돌 사실만 기록한다.
