# 검증 매트릭스 (Verification Matrix)

> 작업 유형별로 **어떤 그라운드 트루스 신호를 붙일지** 매핑하는 표.
> "검증 없는 작업은 시작하지 않는다"(P2)를 실무로 구현하는 도구.
> 이론: `docs/03-loop-engineering.md` 요소1.

## 신호 등급 (신뢰도 순)
| 등급 | 신호 | 의미 |
|------|------|------|
| A | 실행 결과 | 실제 동작을 관찰 (요청/응답, 화면, DB 상태, E2E) |
| B | 테스트 | 단위·통합 테스트 통과 |
| C | 타입/컴파일 | 타입체크·빌드 성공 |
| D | 정적 분석 | 린트·포매터 (보조) |

> 규칙: 각 작업에 **최소 1개**, 가능하면 **여러 등급을 조합**한다. 위험도가 높을수록 상위 등급을 요구.

## 작업 유형별 권장 신호 (채워서 사용)

| 작업 유형 | 권장 신호(등급) | 이 프로젝트의 구체적 명령/방법 |
|-----------|----------------|-------------------------------|
| 순수 함수/유틸 로직 | B + C | `<예: npm test -- utils>` |
| API 엔드포인트 추가/변경 | A + B | `<예: 통합테스트 + curl로 응답 확인>` |
| UI 컴포넌트 | A + C | `<예: 로컬 기동 후 화면 확인 + 타입체크>` |
| 데이터 스키마/마이그레이션 | A + B + 승인 | `<예: 테스트 DB에 적용 후 상태 확인, 사전 승인 필수>` |
| 리팩터링(동작 불변) | B (회귀) | `<예: 기존 테스트 전부 통과 유지>` |
| 버그 수정 | B (재현→회귀) | `<예: 실패 재현 테스트 추가 → 통과 확인>` |
| 성능 개선 | A (측정) | `<예: 벤치마크 전/후 수치 비교>` |
| 설정/인프라 변경 | A + 승인 | `<예: 스테이징 적용 후 헬스체크, 사전 승인>` |

## 검증 게이트 (에이전트에게 강제할 규칙)
1. 변경 후 해당 유형의 신호를 **실행**하고, **출력(증거)** 을 제시한 뒤에만 "완료"라 말한다.
2. 신호가 없거나 만들 수 없는 작업이면 → 착수 전에 신호부터 마련하거나 사람에게 확인한다.
3. 상위 등급(A) 신호가 필요한 위험 작업은 사전 승인 게이트를 통과해야 한다.

## 안티패턴
- 모델의 "완료했습니다"를 신호로 착각 → 금지 (자기 보고는 증거가 아님)
- 테스트를 통과시키려 테스트 자체를 무력화 → 금지 (검증의 목적 훼손)
- 검증 없이 다음 단위로 진행 → 금지 (실패가 누적되어 원인 규명 불가)
